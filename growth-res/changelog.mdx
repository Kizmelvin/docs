The Mono Changelog records **new features**, **API updates**, **deprecations**, and **webhook changes** so you can plan upgrades and keep integrations stable.

## What we document

- **New endpoints or fields** – Added request/response parameters and new APIs
- **Breaking changes** – Deprecations, removed fields, or behaviour changes that may require code updates
- **SDK and library releases** – New versions of official SDKs and any breaking changes
- **Webhook changes** – New or modified webhook events and payloads

Subscribe to release notifications from your [Mono Dashboard](https://app.mono.co/apps) or contact your account manager. For version migration, see the [Migration Guide](/essentials/migration-guide).

## January 2026

### Identity Verification

**TIN Lookup: CAC channel** (31 January)  
The `cac` channel for TIN Lookup is back up and is the most reliable method to verify TIN via NRS. Two new response fields when using the `cac` channel:

- `nrs_tin` – The NRS Tax Identification Number
- `address` – The registered address of the taxpayer

See the [TIN Lookup guide](/lookup/tin-lookup) and [API reference](/api-reference/lookup).

**CAC Lookup: New Status Report Endpoint** (31 January)  
New [CAC Status Report](/api-reference/cac) endpoint returns a company's CAC status report as a PDF.

**CAC Lookup: Exact Search Parameter** (31 January)  
Optional `exact` parameter on [CAC Business Lookup](/api-reference/cac). When `true`, only businesses whose names start with the exact search term are returned. Defaults to `false`.

**Billing notice: BVN Lookup** (22 January)  
All [BVN Lookup API](/lookup/bvn-igree) requests are billable, including failed lookups. Validate BVNs (11-digit numeric) before calling; avoid bulk or blind verification; implement throttling. Contact [support@mono.co](mailto:support@mono.co) if you need BVN Lookup temporarily disabled.

### Financial Data

**New webhook and Account Details fields** (28 January)  
New fields across core webhook events and the Account Details endpoint:

- **app** and **business** added to: `mono.events.account_connected`, `mono.events.account_updated`, `mono.events.account_reauthorized`
- **meta.ref** supported in `mono.events.account_connected` and Account Details API response
- **event_id** (unique) added to: `mono.events.account_connected`, `mono.events.account_reauthorized`, `mono.events.account_updated`

## September 2025

### Financial Data

**data_status and webhook metadata** (16 September)  
Updated Connect behaviour:

- **PARTIAL** (new) – Only one of balance or transactions was retrieved
- **UNAVAILABLE** (new) – Neither retrieved due to bank-related issues or no data
- **FAILED** (updated) – Neither retrieved due to Mono-side issues only

New/updated in API and webhooks: `retrieved_data` (what was fetched), `has_new_data` (boolean in account updated event). `X-HAS-NEW-DATA` header is only sent when new data is available. Update your logic for PARTIAL and UNAVAILABLE; use `retrieved_data` and `has_new_data`. See [Integration guide](/essentials/mono/connect-integration) and [Real-time data](/connect/realtime-data).

### Payments

**Account Debit API: meta and session_id** (22 September)  
[Account Debit API](/api-reference/direct-debit) now supports a `meta` object for custom payment info (returned in response) and `session_id` in API and webhook payloads for tracking.

**Mono Disburse API** (12 September)  
[Mono Disburse](/more-products/disburse) is available: send money to multiple recipients via instant or scheduled transfers from your linked mandate accounts (e.g. payroll, loan disbursements, vendor settlements, cashback).

**Pay with WhatsApp (PWW)** (9 September)  
DirectPay one-time payments support Pay With WhatsApp: customers can complete payment authorisation in the Directpay widget via WhatsApp. No changes required to your Directpay integration.

## January 2025

### Financial Data

**Data Enrichment APIs: job tracker and v2 routes** (31 January)  
New Job Tracker API to monitor Transaction Categorisation, Transaction Metadata, and Statement Insights. New Enrichment Record endpoint to retrieve enriched data by job ID. Deprecated: Transaction Categorisation Records, Transaction Metadata Records, Statement Insights Records. Transaction Categorisation: method changed GET→POST; route to `/v2/accounts/{id}/transactions/categorise`; response includes jobID and jobStatus. CSV Transaction Categorisation: response includes jobID/jobStatus; webhook indicates when categorised data is available (call Enrichment Records for data); data retained 1 hour. Transaction Metadata: GET→POST; route to `/v2/accounts/{id}/transactions/metadata`; same job pattern. CSV Transaction Metadata: same pattern; 1-hour retention. Statement Insights: route to `/v2/accounts/{id}/statement/insights`; webhook indicates when ready (call Enrichment Records); 1-hour retention.
