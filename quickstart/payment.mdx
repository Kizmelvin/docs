Managing recurring payments can be difficult especially if you have to rely on card payments; cards often fail due to insufficient funds or expiry, leading to payment disruptions. These issues create unpredictable cash flow for the business, increase operational costs, and make it harder to maintain a steady revenue stream.

Mono Direct Debit solves this by allowing businesses to collect recurring payments directly from customers' bank accounts on schedule, without using cards. By automating collections and removing manual processing, businesses can reduce payment failures, maintain a steady cash flow, and ensure predictable revenue.

This quick start guide provides clear steps on how to integrate the Mono Direct Debit API into your product, with detailed instructions and sample API requests to help you set up automated payments efficiently.

### Why choose Mono Direct Debit over cards?

Mono Direct Debit ensures more secure payments, as the payment mandate has to be authorised by customer and approved by their bank before the direct debit payment can be initiated by the business.
With lower transaction fees and predictable payment schedules, Mono Direct Debit improves cash flow management and provides a more cost-effective alternative to card payments.
You can read more about the benefits of Direct Debit over cards [here](https://mono.co/blog/cards-vs-direct-debit).

## Prerequisites

To get started, you'll need your Mono secret key and a webhook URL. To get these, follow the steps below:

[Sign up](https://app.mono.co/signup) on the Mono Dashboard and complete KYC.
[Create an App](/quickstart/create-an-app) and set the product to "Payments" to generate your Mono secret key. The Mono secret key is used to make API calls to most of Mono's endpoints, and should be passed in your headers as "mono-sec-key". You can check out this [guide](/quickstart/create-an-app) on how to create an app and obtain your secret key.
Configure a webhook URL and add it to your dashboard. You can use [this site](https://webhook.site/) to quickly generate a webhook URL, and follow this [guide](/utilities/webhooks) to integrate it properly.

  <Step title="Create a mandate">
  This step involves creating a mandate, which allows the customer to authorise your business to to debit their bank account for scheduled payments. To do this, you’ll use the customer ID returned after you created a customer on Mono. .

There are three different <code>MANDATE </code> methods: E-mandate, Signed mandate, and Global Standing Mandate (GSM). You can read more about them in [this guide](/payments/direct-debit-guide).

There are also two <code>DEBIT TYPES </code>, to choose from:

Fixed Debit:
This allows you to collect the same payment amount from customers at set intervals.
Variable Debit:
This allows you to collect varying payment amounts from customers on a custom schedule.
For this guide, we’ll show you how to create an e-mandate with the fixed debit type.

<b>Sample request:</b>

<CodeGroup>
```bash Curl
curl --request POST \
     --url https://api.withmono.com/v2/payments/initiate \
     --header 'accept: application/json' \
     --header 'content-type: application/json' \
     --header 'mono-sec-key: string' \
     --data '
{
    "amount": 100000, // (required) This is the fixed amount to be debited from the user’s account throughout the mandate (i.e from start to end date). NGN 200 (20000 kobo)

    "type": "recurring-debit", // (required) This expects a string value i.e recurring-debit

    "method": "mandate", // (required) This field expects a string value i.e mandate.

    "mandate_type": "emandate", // (required) This can either be signed or emandate or gsm.

    "debit_type": "fixed", // (required) This field expects the type of debit i.e fixed

    "frequency": "monthly", // (required) This expects the frequency of the debit daily , weekly , monthly and yearly. For custom -frequencies use days, weeks, months and provide an whole number value in the interval field

    "retrial_frequency": 1, // (required) This expects the number of retries on a particular debit date [ Minimum is 1 & Maximum is 6 ].

    "initial_debit_amount": 0, // This expects the initial amount to debit on the initial_debit_date

    "initial_date_date" : "2025-03-30", // This expects the first day/date to debit the mandate account. Must be greater than start_date

    "minimum_due": 50000, // (required) This expects an amount in a case of failed debits due to insufficient funds, the minimum_due is attempted i.e anything you can find, give debit

    "grace_period": 2, // (required) This expects the number of days to continue debiting an account before setting the next debit date to next frequency date

    "description": "Subscription payment for movie streaming", // (required) This field expects a description of the set mandate.

    "reference": "subscription1234567890", // (required) This expects a unique reference ID for this particular reference.

    "redirect_url": "https://mono.co", // (required) The webpage or URL the customer should get redirected to after completing the process.

    "customer": {
        "id": "67cac53b271cb661d8e92bc7"
    }, // (required) This expects the customer id

    "start_date": "2025-03-29", // (required) This expects the start date for this mandate to begin. e.g. 2024-12-15

    "end_date": "2028-12-29", // (required) This expects the end date for this mandate to end. e.g. 2024-05-25

    "meta": {}  // This object used to include additional payment information. e.g meta

}'
    ```

## Common issues & debugging tips

Mandate approval delays:

Make sure the NGN 50/100 micro-deposit verification is complete.

Check the events.mandates.created webhook; if the approved field is false, the customer hasn’t completed verification.

Confirm that your webhook URL is correctly set and review your server logs.

Failed recurring debits:

Ensure the customer has sufficient funds on the scheduled debit date.

Check the events.mandates.debit.failed webhook for failure reasons.

Implement retries based on retrial_frequency and confirm that the mandate's start/end dates are valid.

Webhook issues:

Verify the webhook URL and check server logs.

Use ngrok for local debugging.

Check Mono’s webhook logs for delivery status.

**Authentication errors:**

Ensure mono-sec-key header is correct.

Confirm you're using the right secret key (live/test) and update if regenerated.
